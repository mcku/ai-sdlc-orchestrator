# Phase 05 — QA

Verify the change does what it claims, doesn't break anything else, meets the coverage gate, and passes the project's pre-commit hooks.

## Inputs

- `sessions/<id>/03-planning.md` (test strategy)
- Repo state after phase 04

## Outputs

- `sessions/<id>/05-qa.md` containing:
  - Tests added/updated and what they cover
  - Test run results (pass/fail, with output excerpts)
  - **Coverage gate result** (diff coverage %, tool used, any approved exclusions)
  - Regression checks performed
  - UI test results (if applicable)
  - **Pre-commit gate result** (hooks run, outcome, any auto-fixes applied)
  - Known gaps / things not covered

## Required gates (in order)

1. **Tests pass.** Unit tests, plus the agreed regression subset.
2. **Coverage gate** — see `playbooks/coverage-gate.md`. **≥ 80% diff coverage** on added/changed code, using free/open-source tools (`coverage`/`pytest-cov` + `diff-cover`, `c8`, JaCoCo, etc.). Whole-codebase coverage is not the metric.
3. **UI test gate** — if the change touches UI, run the relevant browser/Playwright flows.
4. **Pre-commit gate** — see `playbooks/pre-commit-gate.md`. Run `pre-commit run` against the changed files. Assumes Python and `pre-commit` are installed; if either is missing or `.pre-commit-config.yaml` doesn't exist, record the skip with a reason.

A failure at any gate halts advancement. Resolve, re-run, then proceed.

## Decision points

- Test fails: bug in the change, flaky test, or pre-existing issue surfaced? Triage before patching.
- Coverage gap on added code: add tests, or get explicit user approval to exclude with reason.
- Pre-commit hook fails with auto-fix: re-stage and re-run; auto-fixes become part of the diff and require tests to be re-run.
- Pre-commit hook fails without auto-fix: fix the underlying issue. Do **not** use `--no-verify` and do **not** add per-line exemptions without user approval.
- UI testing required but no browser tooling available: ask the user to install or grant access.

## Tool calls (if available)

- Project test runner.
- Coverage toolchain — see `tools/well-known-tools.md` and `playbooks/coverage-gate.md`.
- `pre-commit run` — see `playbooks/pre-commit-gate.md`.
- Playwright/Browser MCP for UI flows.

## When to ask the user

- Tests that should exist but don't — confirm whether to add them in scope or defer.
- New dev dependencies (coverage tool, `pre-commit`, etc.) before installing.
- Any pre-commit finding you'd like to exempt rather than fix.
- UI testing access (browser MCP, dev server credentials).

## Exit criteria

- All planned tests pass.
- Regression suite (or its agreed subset) passes.
- **Diff coverage ≥ 80%** (or gaps explicitly justified and approved).
- **Pre-commit run is clean** on the changed files (or explicitly skipped with recorded reason).
- `05-qa.md` written.
- Manifest updated with `current_phase: 06-security`.
